# Architecture Guide — W9 Console RPG

How the system is structured, why it evolved this way, and how to extend it.

---

## Week-by-Week Evolution

```
Week 4   IFileHandler           → CSV and JSON file I/O for a single entity type
Week 6   Abstract classes + DIP → Character hierarchy, GameEngine depends on abstractions
Week 7   IContext + FileContext  → Multi-entity data access, combat loop, JSON persistence
Week 9   IContext + GameContext  → EF Core DbContext implements the same IContext contract
```

Each week's code still works. Nothing was thrown away — the architecture grew by adding implementations, not by rewriting what existed.

## The IContext Abstraction

`IContext` is the DIP anchor for all data access. Both `FileContext` and `GameContext` implement it. Business logic (GameEngine, PlayerService, etc.) depends only on `IContext` — never on a concrete context.

```
                        IContext
                       /        \
              FileContext        GameContext
              (JSON files)      (SQL Server / DbContext)
```

### Interface Definition

```csharp
public interface IContext
{
    IEnumerable<Player> Players { get; }
    IEnumerable<MonsterBase> Monsters { get; }
    IEnumerable<Item> Items { get; }
    IEnumerable<Room> Rooms { get; }
    IEnumerable<Character> Characters { get; }

    void AddEntity<T>(T entity) where T : class;
    void RemoveEntity<T>(T entity) where T : class;
    void SaveChanges();
}
```

### Why IEnumerable, Not List or DbSet?

`IEnumerable<T>` is the common ground between `List<T>` (FileContext) and `DbSet<T>` (GameContext). Using it as the return type means consumers don't know or care which backing store they're reading from.

### AddEntity / RemoveEntity

These generic methods let each context handle mutations its own way:
- **FileContext** dispatches by runtime type (`switch` on `Player`, `MonsterBase`, `Item`) and adds to the appropriate `List<T>`
- **GameContext** delegates to EF Core's `Set<T>().Add()` which resolves the correct `DbSet` automatically

## FileContext vs GameContext

| Aspect | FileContext | GameContext |
|--------|-----------|-------------|
| **Backend** | JSON files on disk | SQL Server via EF Core |
| **Storage** | `List<T>` in memory | `DbSet<T>` mapped to tables |
| **Supports** | Players, Monsters, Items | Rooms, Characters |
| **SaveChanges** | Writes `players.json` | Calls `DbContext.SaveChanges()` (SQL) |
| **Read** | `Read()` loads JSON at startup | EF Core queries on demand |
| **Change tracking** | None — manual list ops | EF Core automatic change tracking |

### What Each Context Returns Empty

Neither context covers all five entity types yet:
- **FileContext** returns `Enumerable.Empty<Room>()` and `Enumerable.Empty<Character>()` — no file-backed rooms/characters
- **GameContext** returns `Enumerable.Empty<Player>()`, `Enumerable.Empty<MonsterBase>()`, `Enumerable.Empty<Item>()` — not DB-backed yet

This is by design. Each context owns its domain. A future week could migrate everything to the database.

## How Both Contexts Coexist

`Startup.cs` is the composition root — the only place that creates concrete types:

```csharp
// W7: JSON-backed context for combat features
var fileContext = new FileContext(config.Players, config.Monsters, config.Items);
fileContext.Read();
IContext context = fileContext;

// W9: EF Core context for database features
IContext dbContext = new GameContext();

// Both injected into GameEngine as IContext abstractions
GameEngine = new GameEngine(context, playerService, battleService, GameUi, dbContext);
```

GameEngine has three constructors to support this evolution:

| Constructor | Used By | Parameters |
|------------|---------|------------|
| W6 | `GameEngineDemo` | `(IFileHandler, List<IEntity>)` |
| W7 | (chains to W9) | `(IContext, IPlayerService, IBattleService, IGameUi)` |
| W9 | `Startup` | `(IContext, IPlayerService, IBattleService, IGameUi, IContext?)` |

The W7 constructor chains to W9 with `dbContext: null`, so old code paths still work without a database.

## Namespace Collision: Two Character Classes

The project has two unrelated `Character` classes:

| Class | Namespace | Purpose |
|-------|-----------|---------|
| `Character` | `ConsoleRpg.Models.Characters` | Abstract base class (W6 hierarchy) |
| `Character` | `ConsoleRpgEntities.Models` | EF Core entity (W9 database table) |

GameEngine resolves this with a using alias:

```csharp
using EfCharacter = ConsoleRpgEntities.Models.Character;
```

All W9 methods use `EfCharacter` to avoid ambiguity.

## Room-Character Relationship

One-to-Many: a Room has many Characters, each Character belongs to one Room.

```
Room (1) ───── (*) Character
  Id                 Id
  Name               Name
  Description        Level
  Characters ◄──── RoomId (FK)
                     Room (navigation)
```

EF Core configures this relationship by convention — `RoomId` is the foreign key, `Room` is the navigation property. `Include(c => c.Room)` in `DisplayCharacters()` does eager loading to avoid lazy-loading pitfalls.

## SOLID Principles in Action

### Single Responsibility (SRP)
- `Program.cs` — entry point and menu loop only
- `Startup.cs` — dependency wiring only
- `ConsoleGameUi` — display and input only
- `GameEngine` — business logic only
- `FileContext` / `GameContext` — data access only

### Open/Closed (OCP)
- Adding a new data backend means implementing `IContext` — no existing code changes
- Adding a new entity to FileContext means adding a list, updating `Read/AddEntity/RemoveEntity/SaveChanges`

### Liskov Substitution (LSP)
- `FileContext` and `GameContext` are interchangeable through `IContext`
- `CsvFileHandler` and `JsonFileHandler` are interchangeable through `IFileHandler`

### Interface Segregation (ISP)
- `IContext`, `IFileHandler`, `IPlayerService`, `IBattleService`, `IGameUi` — each interface serves one consumer's needs

### Dependency Inversion (DIP)
- Every consumer depends on abstractions (`IContext`, `IPlayerService`, etc.)
- Only `Startup.cs` knows about concrete types
- GameEngine has zero `new` calls for its dependencies

## How to Add a New Entity

Step-by-step for adding a new EF Core entity (e.g., `Weapon`):

1. **Create the model** in `ConsoleRpgEntities/Models/Weapon.cs`
   - Add `Id`, `Name`, and any other properties
   - Add a foreign key if it relates to another entity (e.g., `CharacterId`)

2. **Add a DbSet** to `GameContext.cs`
   ```csharp
   public DbSet<Weapon> Weapons { get; set; }
   ```

3. **Add to IContext** (if business logic needs it)
   ```csharp
   IEnumerable<Weapon> Weapons { get; }
   ```
   Update `FileContext` to return `Enumerable.Empty<Weapon>()`.

4. **Generate a migration**
   ```bash
   dotnet ef migrations add AddWeapon --project ConsoleRpgEntities --startup-project ConsoleRpg
   ```

5. **Apply the migration**
   ```bash
   dotnet ef database update --project ConsoleRpgEntities --startup-project ConsoleRpg
   ```

6. **Add GameEngine methods** for CRUD operations on the new entity.

7. **Add menu options** in `ConsoleGameUi.GetMenuChoice()` and `Program.cs`.

## Key Design Decisions

### Why not replace IContext entirely with DbContext?

W7 combat features depend on JSON files (players, monsters, items). Ripping out FileContext would break working features for no gain. Instead, both contexts coexist — each owns its domain. This is OCP: extend by addition, not modification.

### Why explicit interface implementation for SaveChanges?

`DbContext.SaveChanges()` returns `int` (rows affected). `IContext.SaveChanges()` returns `void`. The explicit implementation bridges the two:

```csharp
void IContext.SaveChanges()
{
    base.SaveChanges();  // calls DbContext.SaveChanges(), discards the int
}
```

### Why IEnumerable instead of IQueryable?

`IQueryable` would leak EF Core details into business logic (expression trees, deferred SQL execution). `IEnumerable` keeps the abstraction clean. Where EF-specific features are needed (like `Include()`), GameEngine casts locally — the leak is contained to one method, not the interface.

### Why a using alias instead of renaming?

Both `Character` classes are well-named for their contexts. Renaming either would reduce clarity. The alias (`using EfCharacter = ...`) is local to GameEngine and self-documenting.
